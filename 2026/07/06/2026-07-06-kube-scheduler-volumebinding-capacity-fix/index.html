Kubernetes volumebinding 调度插件容量检查修复全记录

Posted on 2026-07-06 In Cloud

在排查 local-dynamic StorageClass 导致的 Pod 调度异常时，逐步发现并修复了 volumebinding 插件中三个独立的容量检查 Bug。本文完整记录从现象、根因、日志实证到修复方案的全过程。

问题背景

在节点上同时创建多个 Pod + PVC（使用 local-dynamic StorageClass，节点容量恰好满足 N 个 PVC），出现两类互相矛盾的现象：

一个 Pod 被拒绝调度，另一个 Pod 被调度到节点后 Provisioner 报空间不足——同一个节点，调度器时而过于严格，时而过于宽松。

三个 Bug 的修复总览

Bug 根因 现象 修复
Bug A（双重计算） PV informer 早于 PVC VolumeName informer 到达（实测 161ms） pod-10 在窗口内被错误拒绝 改用 AnnSelectedNode 扫描为唯一来源（0099 patch）

总结

local-dynamic PVC 调度异常（过量调度 + 错误拒绝）由多个独立 Bug 共同造成：

  1. pvCache + pod-walk 的双来源缺陷（0099 patch）：PV informer 和 PVC informer 各自延迟，产生双重计算窗口；PreBind 失败后 ForgetPod 导致容量预约完全消失。修复：以 AnnSelectedNode 注解为单一可靠来源，彻底消除两个方向的竞态。

  2. Released PV 漏计（0099 patch）：Retain 策略下已删除 PVC 对应的 PV 仍占用节点空间，但 AnnSelectedNode 扫描无法发现。修复：增加专项 Released PV 扫描。

  3. className 非确定性 + Provisioner 范围盲区（v1.34.3 branch）：continue 而非 break 导致 className 被最后一个匹配的 StorageClass 覆盖；单 className 查询无法覆盖共享同一 Provisioner 的多个 StorageClass。修复：收集 classNamesForDriver 集合，所有来源统一按 Provisioner 范围过滤。

核心洞察：容量预约的真实范围是 Provisioner，而非 StorageClass。多个 StorageClass 可以共享同一个 Provisioner 和同一块物理存储池，必须将它们整体视为一个容量域。
