Virtlet、Kata Containers 与 KubeVirt 核心原理对比

Posted on 2026-07-08 Edited on 2026-07-09 In Cloud

在 Kubernetes 生态中，有三种主流的 VM/容器融合方案：Virtlet、Kata Containers 和 KubeVirt。本文对比它们的核心原理与适用场景。

Tap Device vs Domain Socket

首先澄清一个常见疑问：tap device 不是 domain socket，两者是完全不同的机制。

Tap Device Unix Domain Socket
本质 内核虚拟网络接口（Layer 2） 进程间通信端点
存在形式 网卡接口（如 tap0） 文件系统路径（如 /var/run/foo.sock）
用途 VM 网络通信 IPC（进程间通信）

Kata Containers 核心原理

一句话：每个容器/Pod 跑在一个轻量级 VM 里，但对外表现得像普通容器。

架构层次

关键组件

  • 轻量级 Hypervisor：默认 QEMU，也支持 Cloud Hypervisor、Firecracker，启动极快（~100ms）
  • kata-agent：跑在 VM 内部的 gRPC server，接收宿主机指令在 VM 内执行容器操作
  • virtio 设备：
    • virtio-fs / virtio-9p：挂载容器 rootfs 进 VM
    • virtio-net + tap device：网络通信
    • vhost-vsock：宿主机 ↔ VM 内 kata-agent 通信

隔离原理

每个 Pod 有自己独立的 Guest Kernel，安全隔离远强于普通容器。

架构对比

Virtlet（已废弃）

直接实现 CRI，kubelet 以为在管容器，实际跑的是 VM。绕过 containerd，维护成本高，社区已放弃。

KubeVirt

  • VM 是独立 CRD，不伪装成容器
  • 完整 VM 语义：快照、热迁移、设备直通
  • virt-launcher 本身是个 Pod，VM 跑在 Pod 里

Kata Containers

用户用的还是普通容器镜像，感知不到 VM，每个 Pod = 一个轻量 VM。
