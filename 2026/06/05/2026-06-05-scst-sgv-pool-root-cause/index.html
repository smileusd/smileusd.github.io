SCST Infinite Loop: The Real Root Cause Was a Missing SGV Pool Flush

Background

In the previous post we
analyzed an infinite loop in the SCST kernel module’s device cleanup path and produced a
working workaround (diagnostics + force-cleanup after 1,000 iterations). This post
documents what instrumentation revealed about the actual root cause and shows that
the real fix is just two lines of code.

What Instrumentation Revealed

After adding per-ucmd logging inside dev_user_unjam_dev(), the picture became clear:

Three facts stand out:

  1. Always the same ucmd pointer — 00000000ff3afdd1 — across all 2 million iterations
    per second. It never changes.
  2. state = 7 = UCMD_STATE_ON_FREE_SKIPPED, scst_cmd = NULL, ucmd_ref = 1.
  3. Forcing ucmd_ref to 0 does not break the loop. Even after the test code set the
    atomic to zero, the ucmd pointer remained in ucmd_hash and the loop continued.

That third observation is the most important: it proves that the issue is not a
reference-count value problem. The ucmd is stuck in the hash because
dev_user_free_ucmd() — the function that calls cmd_remove_hash() — was never
invoked. Setting a raw atomic to zero bypasses ucmd_put() and its
atomic_dec_and_test() trigger, so nothing ever removes the entry from the list.

State 7: UCMD_STATE_ON_FREE_SKIPPED

There is exactly one place in the codebase that sets this state:

dev_user_process_reply_on_free is:

At first glance this looks correct: free the SGV memory, drop the reference, done. So why
does the ucmd stay in the hash?

The SGV Pool Is a Cache, Not a Direct Free

The answer is in dev_user_free_sgv():

The call is sgv_pool_free(), not a direct page free. The SGV (scatter-gather vector)
pool is a performance cache: it holds recently freed SG buffers so that future
commands can reuse them without going to the page allocator. When sgv_pool_free() is
called:

  • The SGV object is placed on the pool’s LRU cache.
  • The allocator’s free callback — dev_user_free_sg_entries() — is not called.
  • The pages are not freed.
  • The ucmd_get() reference taken in dev_user_alloc_pages() is not released.

dev_user_free_sg_entries() (and therefore its ucmd_put()) only fires when the pool
evicts a cached object, either because the pool is purged or because
sgv_pool_flush() is called explicitly.

The Complete Reference Count Trace

Here is the exact reference sequence for a command being unjammed during device teardown:

Event Operation ucmd_ref
dev_user_alloc_ucmd() atomic_set(&ucmd_ref, 1) 1
dev_user_alloc_pages() first page ucmd_get() 2
dev_user_unjam_dev(): ucmd_get_check() atomic_inc_return() → 3, ≠ 1 → not zombie 3
dev_user_unjam_cmd() → scst_cmd_done() → dev_user_on_free_cmd() → dev_user_process_reply_on_free(): dev_user_free_sgv() → sgv_pool_free() SGV goes to pool cache; dev_user_free_sg_entries() not called; alloc_pages ref not released 3
dev_user_process_reply_on_free(): ucmd_put() atomic_dec_and_test() → 2, ≠ 0 2
Back in dev_user_unjam_dev(): ucmd_put() for ucmd_get_check ref atomic_dec_and_test() → 1, ≠ 0 1

atomic_dec_and_test() returns true — and therefore calls dev_user_free_ucmd() →
cmd_remove_hash() — only when the result is 0. It never reaches 0 because the
alloc_pages reference (held by dev_user_alloc_pages() via ucmd_get()) is never
released. dev_user_free_sg_entries() never fires. The ucmd stays in the hash forever.

Why the Loop Runs at 2 Million Iterations per Second

After the unjam pass the ucmd has:

  • sent_to_user = 0 (reset by dev_user_unjam_cmd() before calling scst_cmd_done)
  • state = UCMD_STATE_ON_FREE_SKIPPED
  • cmd = NULL
  • not in ready_cmd_list

On every subsequent call to dev_user_unjam_dev():

res is incremented (non-zero) but the command is not unjammed. dev_user_get_next_cmd()
returns -EAGAIN because the ucmd is not in ready_cmd_list. With
dev->cleanup_done = 1, the outer loop in dev_user_process_cleanup() has no exit:

Both dev_user_unjam_dev() and dev_user_get_next_cmd() acquire and release a spinlock
and complete in under a microsecond. Result: ~2 million iterations per second, 100% CPU
on one core, kernel soft-lockup detector fires within seconds.

Why the Previous Analysis Was Incomplete

The earlier post described the problem as “zombie commands (refcount=0) that
dev_user_unjam_dev() still counts in res++“. That is a real code smell — counting
before checking — but it is not the root cause in this case.

The stuck ucmd has ref=1, not ref=0. ucmd_get_check() succeeds (ref goes 1→2→1),
meaning the command is not a zombie. The unjam loop simply skips it because
sent_to_user=0. The ref never reaches 0 because the SGV pool holds onto the buffer
without calling the free callback.

The Fix: Two Lines After the Unjam Loop

Why this works:

sgv_pool_flush() forces the pool to evict all cached objects. For each evicted object,
the registered free callback fires:

On the next call to dev_user_unjam_dev(), the hash is empty, res=0, and
dev_user_process_cleanup() breaks out of its loop normally.

Why the pre-loop flush was not enough:

The existing flush at the top of dev_user_unjam_dev() runs before the spinlock is
acquired, before any command is unjammed. SGV objects are only placed into the cache
during unjamming (step 2 in the trace above). Those objects are invisible to the
pre-loop flush.

With the SGV flush fix in place, the loop terminates within 2–3 iterations. The

triggered by the flush have CPU time to run before the next iteration.

Lessons

1. Pool caches hide reference lifetimes

The SGV pool is an optimization that decouples sgv_pool_free() from the actual page
release. Code that relies on “free → callback → ucmd_put” must account for the
possibility that the callback fires asynchronously (on eviction), not synchronously (on
free).

2. Instrumentation before conclusion

The earlier analysis correctly identified the symptoms (infinite loop, stuck ucmd) and
the code smell (counting before checking in dev_user_unjam_dev). But the true cause
required observing the actual ucmd_ref value at runtime. Adding that single log line
— showing ref=1 instead of the assumed ref=0 — immediately pointed to a missing
ucmd_put, which led straight to the SGV pool caching behavior.

3. Setting an atomic to zero is not the same as ucmd_put

The test that forced ucmd_ref to 0 and observed the loop continuing was the key
falsification experiment. It proved that the issue was not about the reference value
but about the path that calls dev_user_free_ucmd(). No matter what the counter
shows, nothing removes the ucmd from the hash unless atomic_dec_and_test() returns
true inside ucmd_put().

4. Minimal fixes beat complex ones

The final fix is two sgv_pool_flush() calls added after the loop — about 5 lines
including the comment. The previous workaround was 80+ lines of diagnostic logging,
force-cleanup, and msleep throttling, none of which addressed the root cause.

Summary

Detail
Symptom dev_user_process_cleanup() loops at 2M iterations/second; soft lockup
Stuck ucmd state=7 (ON_FREE_SKIPPED), scst_cmd=NULL, ref=1, not in ready list
Root cause sgv_pool_free() caches the SGV object instead of freeing it; dev_user_free_sg_entries() never fires; alloc_pages ucmd_get() reference never released
Why flush order matters Pre-loop flush runs before unjamming; SGV objects cached during unjamming are never evicted
